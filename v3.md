# Symposium 3.0 — Refactor Plan

Major release. Breaking changes are allowed.

## Goals

1. Replace the `EventEmitter`-based `Agent.message()` contract with an **async generator**, so consumers can iterate events naturally with `for await`.
2. Accept **streaming input** (`AsyncIterable<string | ContentBlock | ControlMessage>`) in addition to plain `string` / `ContentBlock[]`, enabling concurrent user messages mid-generation.
3. Make the model layer **actually stream** (today `Model.generate()` awaits the full response and synthesizes fake "chunks").
4. Clean up the chat-vs-utility return-type asymmetry and add structured output for chat agents.
5. Replace the callback-style tool authorization (`confirmFunctions`) with a control message flowing through the input channel.

## Assessment of current code

- `Agent.message(content, thread)` returns a `BufferedEventEmitter` for chat agents, but the resolved Promise value for utility agents (`Agent.js:168-207`, `Agent.js:315`) — same method, two return types.
- `BufferedEventEmitter` exists purely to buffer events until listeners attach — a workaround for the race between returning the emitter and the consumer adding handlers (`BufferedEventEmitter.js:6-12`). Will be deleted in 3.0.
- `OpenAIModel.generate()` awaits the full `responses.create()` and synthesizes events from the finished response (`OpenAIModel.js:143-197`). Other model adapters likely the same. The "stream of events over time" the emitter implies is half-fictional today.
- Internal turn loop is recursive: `execute()` → `handleCompletion()` → `callFunctions()` → `{type:'continue'}` → `execute()` again (`Agent.js:215-316`). Will collapse to a `while` loop inside the generator.
- Tool authorization currently emits `tools_auth` with `{thread, functions, completion, emitter}` and expects the consumer to call `agent.confirmFunctions(...)` to resume (`Agent.js:455-459`, `Agent.js:479`).

## Decisions taken

| Question | Decision |
| --- | --- |
| When a user message arrives mid-generation | Wait for current tool call to finish, then insert the new user message before the next LLM turn. No `AbortSignal` plumbing through tools. |
| How to surface retries | Hybrid: silent retry if no `output` event has been yielded yet (common case — connection errors before any tokens); visible `retry` event once any output has been streamed. |
| Input stream API | Plain `AsyncIterable<string \| ContentBlock \| ControlMessage>` accepted directly. Ship a `createInputChannel()` helper for the common "push messages from elsewhere" use case. `agent.message('hello')` (plain string) still works. |
| Tool authorization | Yield a `tools_auth` event, then wait for an `{type: 'auth', id, decision}` control message on the input channel. No more `confirmFunctions` API. |
| Chat vs utility return type | `utility` agents: `await agent.message(...)` returns the parsed value directly (no generator). `chat` agents: always return an async generator. Structured output becomes an independent capability (`response_schema`) usable by either. |

## Event shape (chat agents)

Discriminated union, all events yielded from the generator:

```
{type: 'start',         thread}
{type: 'output',        content}              // text chunk or image
{type: 'reasoning',     content}
{type: 'tool',          name, arguments, id}
{type: 'tool_response', name, success, response?, error?}
{type: 'tools_auth',    id, functions}
{type: 'retry',         attempt, reason}
{type: 'result',        value}                // only if response_schema is set
{type: 'end',           thread}
```

Errors throw out of the generator (no more `error` event).

## Input channel control messages

In addition to user-message content, the input iterable may yield control objects:

```
{type: 'auth',   id, decision: 'approve' | 'approve_always' | 'reject'}
{type: 'submit'}                  // optional: terminates the initial user message build-up
{type: 'cancel'}                  // gracefully stops the agent loop
```

Closing the iterable is equivalent to `cancel` once the in-flight turn completes.

## Phases

### Phase 1 — Real streaming in the model layer

**Files:** `Model.js`, `Models/OpenAIModel.js`, `Models/AnthropicModel.js`, `Models/GroqModel.js`, `Models/DeepSeekModel.js`, `Models/GrokModel.js`, `Models/OllamaModel.js`, `Models/LegacyOpenAIModel.js`.

- Change `Model.generate()` contract from `→ Promise<Message[]>` to `async function*` yielding deltas:
  - `{type: 'text_delta', content}`
  - `{type: 'reasoning_delta', content}`
  - `{type: 'tool_call', content}` (emitted complete — tool-call deltas are rarely useful to consumers)
  - `{type: 'image', content, meta}`
  - Final `return` value: the assembled `Message[]` (so callers that need the full thing still get it via `generator.return()` semantics or by accumulating).
- Switch OpenAI/Anthropic/Groq adapters to their streaming APIs (`stream: true` / `responses.stream()` / `messages.stream()`).
- For models without a streaming SDK (Ollama if applicable), emulate by yielding the whole text once.

This unblocks everything downstream and is internally testable.

### Phase 2 — Convert `Agent` to async generators

**Files:** `Agent.js`. Delete `BufferedEventEmitter.js`.

- `Agent.message(content, thread)` becomes `async function*`.
- The recursive `execute()` chain collapses into a single `while (true)` inside the generator: pull a model completion, yield events, run tools, loop until no more tool calls.
- Errors throw out of the generator naturally. No more `error` event.
- `BufferedEventEmitter` buffering hack disappears.

After this phase the input parameter is still `string | ContentBlock[]` only.

### Phase 3 — Streaming input

**Files:** `Agent.js`, new `InputChannel.js` helper.

- Widen the `content` parameter to `string | ContentBlock | ContentBlock[] | AsyncIterable<string | ContentBlock | ControlMessage>`.
- Strings/arrays continue to work — internally wrapped in a trivial single-shot iterable.
- For an `AsyncIterable` input:
  1. Drain the iterable into the initial user message until a `submit` marker arrives, OR until the iterable closes (whichever first). If neither, drain until the iterable has yielded at least one piece of content and then start the first model turn.
  2. Begin the model loop.
  3. **Concurrently** keep reading from the input iterable. On each tool-loop boundary (between LLM turns), check for newly arrived user messages and insert them as `user` messages before the next LLM call. Mid-tool-call messages wait — per the decision above.
  4. When the input iterable closes AND no model work is in flight, the generator finishes.
- Ship `createInputChannel()` returning `{send(item), close(), [Symbol.asyncIterator]()}` for the common "I need to push messages from elsewhere" case. Implemented with a simple promise-based queue.
- Retry buffering: items already consumed from the input iterable get committed to the thread before the model call, so retries don't need to re-read the source.

### Phase 4 — Tool auth via the input channel

**Files:** `Agent.js`. Remove `confirmFunctions(...)`.

- Replace the `confirmFunctions(...)` callback API.
- On auth required: yield `{type: 'tools_auth', id, functions: [...]}`, then `await` an `{type: 'auth', id, decision}` control message from the input iterable.
- Consumer flow:
  ```js
  const input = createInputChannel();
  for await (const ev of agent.message(input)) {
    if (ev.type === 'tools_auth') {
      const decision = await askUser(ev.functions);
      input.send({type: 'auth', id: ev.id, decision});
    }
  }
  ```
- If the input channel closes without an auth response, treat it as reject + cancel.

### Phase 5 — Retry strategy (hybrid)

**Files:** `Agent.js`.

- Track an `output_yielded` flag inside the generator.
- Errors during `generateCompletion()`:
  - If `!output_yielded` and `attempt < max_retries`: silently retry (existing behavior).
  - If `output_yielded` and `attempt < max_retries`: yield `{type: 'retry', attempt, reason}`, then retry.
  - If exhausted: throw.
- Errors during tool execution stay where they are (caught and routed into `tool_response.error`).

### Phase 6 — Utility / structured output split

**Files:** `Agent.js`, `Symposium.js`.

- Decouple two concerns:
  - `type: 'chat' | 'utility'` → return shape (generator vs. value).
  - `response_schema` (new, optional) → if set, the final assistant turn must conform.
- Behavior:
  - `utility` agents: `await agent.message(...)` returns the parsed value directly. No generator surface. Internally still uses the same model streaming, just drains it.
  - `chat` agents with `response_schema`: generator yields normal events, and the final event is `{type: 'result', value: <parsed>}`. Useful for the "kinda utility" case (tools first, then structured answer).
  - `chat` agents without `response_schema`: no `result` event, behavior as today.
- Move the existing `utility.type/function/parameters` config under `response_schema` for clarity. Keep a back-compat shim that maps the old shape for one minor version, then drop it.

### Phase 7 — Docs, examples, migration guide, version bump

**Files:** `README.md`, `package.json`, new `MIGRATION.md`.

- Rewrite README for the generator API and the input channel.
- Migration guide with side-by-side before/after for the most common patterns:
  - Simple chat
  - Tool use
  - Tool authorization
  - Structured output
  - Streaming user input
- Bump `package.json` to `3.0.0`.

---

## Phase ordering

Phases 1 → 2 → 3 → 4 are sequential (each depends on the previous). Phases 5 and 6 can land any time after Phase 2 and are independent of each other. Phase 7 is last.

## Open considerations

- **No live API testing in the dev environment.** The streaming model adapters (Phase 1) cannot be end-to-end tested here without API keys / network access in a way that exercises real provider behavior. Validation will need to happen on the maintainer's side. Mocked-stream unit tests are still worthwhile.
- **No test suite exists today.** Worth adding at least a few smoke tests alongside the refactor — happy path for chat, tool loop, structured output, streaming input.
- **Backpressure.** With generators, a slow consumer pauses the model upstream. This is generally an improvement over fire-and-forget emitter events, but it's a behavioral change worth noting in the migration guide.
- **Multi-consumer.** Async generators are single-consumer. If a use case needs multi-cast (two listeners on the same agent run), the consumer needs to `tee` manually. Not expected to be common.

## Target API (informative)

```js
// Simple chat — generator
for await (const ev of agent.message('Hello')) {
  if (ev.type === 'output') process.stdout.write(ev.content);
}

// Streaming input + tool auth
const input = createInputChannel();
input.send('Plan a trip to Rome');
const events = agent.message(input);

for await (const ev of events) {
  switch (ev.type) {
    case 'output':       process.stdout.write(ev.content); break;
    case 'tools_auth':   input.send({type: 'auth', id: ev.id, decision: 'approve'}); break;
    case 'retry':        console.warn(`retrying (${ev.attempt}): ${ev.reason}`); break;
  }
}

// Concurrently, from elsewhere:
input.send('Actually, make it Florence instead');

// Utility — direct value
const data = await extractor.message('My name is John, email john@example.com');
//   → { name: 'John', email: 'john@example.com' }

// Chat with structured final answer
for await (const ev of agent.message('Look up the weather and reply in JSON')) {
  if (ev.type === 'result') console.log(ev.value); // schema-conforming object
}
```
